Background

{{The Universal Asynchronous Receiver-Transmitter (UART) is a serial communication protocol used for peer-to-peer data transfer. Unlike I2C/SPI, it doesn't need a shared clock—it relies on pre-negotiated baud rates.}}

Example (UART): The Universal Asynchronous Receiver-Transmitter (UART) is a foundational serial communication protocol used in almost every embedded system. Unlike synchronous protocols (I2C/SPI), UART does not require a shared clock line—it relies on pre-negotiated baud rates and precise timing. This makes it ideal for long-distance communication between microcontrollers, GPS modules, and Bluetooth chips.

In this project, I designed a modular UART transceiver supporting the 8N1 format (8 data bits, No parity, 1 stop bit). The design is fully synthesizable and validated with a self-checking testbench.

Why this matters: UART is the "hello world" of serial communication. Mastering it proves you understand timing, FSM design, and cycle-accurate simulation—core skills for any RTL engineer.

Principle & Architecture

{{The design has 3 blocks: Baud Generator (divides clock), Transmitter (FSM: IDLE→START→DATA→STOP), and Receiver (detects START, samples at mid-bit using 16x oversampling).}}

Example (UART): The top-level design consists of three main blocks:

  • Baud Generator: Divides the system clock (100 MHz) down to the target baud rate (115200 Hz) using a programmable counter.
  • Transmitter (TX): A Finite State Machine (FSM) that idles HIGH. On a load signal, it pulls the line LOW (START bit), shifts out 8 data bits (LSB-first), and then returns HIGH (STOP bit).
  • Receiver (RX): A more complex FSM that detects the START bit (HIGH→LOW transition), oversamples at 16x the baud rate to sample each bit in the center (noise rejection), and asserts a data_ready flag after receiving the STOP bit.
The receiver uses a mid-bit sampling technique to avoid glitches on the line, making it robust in noisy environments.

State Machine Flow: IDLE → START → DATA[0:7] → STOP → IDLE.

Usage in VLSI / Silicon

{{UART is embedded as an AXI/AHB peripheral. Key VLSI concerns: Clock Domain Crossing (CDC) synchronization, power-gating for wake-on-UART, and timing closure for the baud counter.}}

Example (UART): In modern System-on-Chip (SoC) designs, UART is embedded as a peripheral bridge between the high-speed processor bus (e.g., AXI/AHB) and low-speed external devices.

Key VLSI considerations include:

  • Clock Domain Crossing (CDC): The UART typically operates in a separate frequency domain. A synchronizer is often added to the RX line to prevent metastability.
  • Low-Power Design: UART peripherals are often power-gated in mobile SoCs and woken up only when a START bit is detected (wake-on-UART).
  • Synthesis constraints: The baud generator must meet timing even at high clock frequencies (e.g., 200 MHz). Careful counter logic ensures no timing violations.

Real-World Applications

{{IoT sensors (GPS/MPU6050), industrial PLC communication, USB-to-serial debug bridges (FTDI), automotive OBD-II diagnostic ports.}}

Example (UART): UART is everywhere in embedded electronics:

  • IoT Sensors: Interfacing ESP32/Arduino with GPS, MPU6050 (IMU), and environmental sensors—directly connects to my MotoScore and SVASTI projects.
  • Industrial Automation: Communication between Programmable Logic Controllers (PLCs) and legacy serial terminals (RS-232/RS-485).
  • Debugging: UART-to-USB bridges (e.g., FTDI, CP2102) are the standard method for firmware logs and bootloader communication in almost every embedded board.
  • Automotive: Used in OBD-II diagnostic ports for vehicle telemetry.

Below is the synthesizable RTL for the top-level module. The code follows a clean, modular structure.

uart_tx.v Synthesizable

        // ------------------------------------------------------------
        // Module: uart_tx
        // Description: 
        //   {{Top-level wrapper integrating baud_gen, transmitter, and receiver. Supports 8N1 protocol at 115200 baud.}}
        //   Designed as part of the 100-day VLSI journey.
        // Performance: Operates at {{BAUD_RATE}} baud. 
        //   Latency: 1 cycle load + 10 bit-periods for serialization.
        // ------------------------------------------------------------
        module uart_tx (
            input wire clk,        // System clock (e.g., 100 MHz)
            input wire rst_n,      // Active-low reset
            input wire [7:0] tx_data,
            input wire tx_en,
            output wire tx_out,
            input wire rx_in,
            output wire [7:0] rx_data,
            output wire rx_valid
        );
        
            // Wire declarations for internal connections
            wire clk_baud;
            wire tx_busy;
        
            // Instantiate Baud Generator
            baud_gen #(.DIVISOR({{DIVISOR_VALUE}})) u_baud (
                .clk(clk),
                .rst_n(rst_n),
                .baud_clk(clk_baud)
            );
        
            // Instantiate Transmitter
            uart_tx u_tx (
                .clk(clk_baud),
                .rst_n(rst_n),
                .data_in(tx_data),
                .tx_en(tx_en),
                .tx_busy(tx_busy),
                .tx_out(tx_out)
            );
        
            // Instantiate Receiver
            uart_rx u_rx (
                .clk(clk_baud),
                .rst_n(rst_n),
                .rx_in(rx_in),
                .data_out(rx_data),
                .rx_valid(rx_valid)
            );
        
        endmodule

Self-Checking Testbench

A self-checking testbench is critical for automated verification. It sends 5 random test vectors through the UART (TX → RX loopback) and compares the output without manual waveform inspection.

tb_{{uart_tx}}.v Self-Checking 
uart_tx_tb_code.png
        // ------------------------------------------------------------
        // Testbench: 
        // Features:
        //   - Loopback connection (tx_out to rx_in)
        //   - Generates 5 random 8-bit values
        //   - Automatically compares sent vs received data
        //   - Prints "PASS" or "FAIL" to the console
        //   - Dumps .vcd file for GTKWave analysis
        // ------------------------------------------------------------
        `timescale 1ns / 1ps
        
        module tb_uart_tx;
        
            reg clk;
            reg rst_n;
            reg [7:0] tx_data;
            reg tx_en;
            wire tx_out;
            wire [7:0] rx_data;
            wire rx_valid;
            wire rx_in;
        
            // DUT Instantiation
            {{TOP_MODULE_NAME}} dut (
                .clk(clk),
                .rst_n(rst_n),
                .tx_data(tx_data),
                .tx_en(tx_en),
                .tx_out(tx_out),
                .rx_in(rx_in),
                .rx_data(rx_data),
                .rx_valid(rx_valid)
            );
        
            // Loopback connection
            assign rx_in = tx_out;
        
            // Clock generation (10ns period -> 100MHz)
            always #5 clk = ~clk;
        
            // Test Sequence
            initial begin
                $dumpfile("dump.vcd");
                $dumpvars(0, tb_{{TOP_MODULE_NAME}});
        
                // Initialize
                clk = 0;
                rst_n = 0;
                tx_data = 8'h00;
                tx_en = 0;
                #20 rst_n = 1;
        
                // Test 5 random values
                repeat(5) begin
                    tx_data = $random;        // Generate random byte
                    tx_en = 1;
                    #20 tx_en = 0;
                    #200;                     // Wait for transmission to complete
                    
                    // Check if received data matches sent data
                    if (rx_valid && (rx_data == tx_data))
                        $display("[PASS] Sent: %h, Received: %h", tx_data, rx_data);
                    else
                        $display("[FAIL] Sent: %h, Received: %h", tx_data, rx_data);
                end
                
                $display("Simulation finished.");
                #100 $finish;
            end
        
        endmodule
