Background

The Universal Asynchronous Receiver-Transmitter (UART) is a foundational serial communication protocol used in almost every embedded system. Unlike synchronous protocols (I2C/SPI), UART does not require a shared clock line—it relies on pre-negotiated baud rates and precise timing. This makes it ideal for long-distance communication between microcontrollers, GPS modules, and Bluetooth chips.

In this project, I designed a modular UART transceiver supporting the 8N1 format (8 data bits, No parity, 1 stop bit). The design is fully synthesizable and validated with a self-checking testbench.

Why this matters: UART is the "hello world" of serial communication. Mastering it proves you understand timing, FSM design, and cycle-accurate simulation—core skills for any RTL engineer.

Principle & Architecture

The top-level design consists of three main blocks:

  • Baud Generator: Divides the system clock (100 MHz) down to the target baud rate (115200 Hz) using a programmable counter.
  • Transmitter (TX): A Finite State Machine (FSM) that idles HIGH. On a load signal, it pulls the line LOW (START bit), shifts out 8 data bits (LSB-first), and then returns HIGH (STOP bit).
  • Receiver (RX): A more complex FSM that detects the START bit (HIGH→LOW transition), oversamples at 16x the baud rate to sample each bit in the center (noise rejection), and asserts a data_ready flag after receiving the STOP bit.
The receiver uses a mid-bit sampling technique to avoid glitches on the line, making it robust in noisy environments.

State Machine Flow: IDLE → START → DATA[0:7] → STOP → IDLE.

Usage in VLSI / Silicon

In modern System-on-Chip (SoC) designs, UART is embedded as a peripheral bridge between the high-speed processor bus (e.g., AXI/AHB) and low-speed external devices.

Key VLSI considerations include:

  • Clock Domain Crossing (CDC): The UART typically operates in a separate frequency domain. A synchronizer is often added to the RX line to prevent metastability.
  • Low-Power Design: UART peripherals are often power-gated in mobile SoCs and woken up only when a START bit is detected (wake-on-UART).
  • Synthesis constraints: The baud generator must meet timing even at high clock frequencies (e.g., 200 MHz). Careful counter logic ensures no timing violations.

Real-World Applications

UART is everywhere in embedded electronics:

  • IoT Sensors: Interfacing ESP32/Arduino with GPS, MPU6050 (IMU), and environmental sensors—directly connects to my MotoScore and SVASTI projects.
  • Industrial Automation: Communication between Programmable Logic Controllers (PLCs) and legacy serial terminals (RS-232/RS-485).
  • Debugging: UART-to-USB bridges (e.g., FTDI, CP2102) are the standard method for firmware logs and bootloader communication in almost every embedded board.
  • Automotive: Used in OBD-II diagnostic ports for vehicle telemetry.

Self-Checking Testbench

A self-checking testbench is critical for automated verification. It sends random test vectors through the UART (TX → RX loopback) and compares the output without manual waveform inspection.

Transmitter Testbench
tb_uart_transmitter.v Self-Checking
Validates TX FSM transitions and bit timing. Generates .vcd dump for waveform analysis.
Receiver Testbench
tb_uart_receiver.v Self-Checking
Validates START detection, mid-bit sampling, and data integrity. Prints PASS/FAIL to console.
Run Command: iverilog -o sim uart_tx.v uart_rx.v tb_uart_top.v && vvp sim && gtkwave dump.vcd
